# Solution: [Name]

This profile defines how to message a specific solution — a real, outcome-oriented use case where the portfolio delivers measurable value. Solutions bridge the gap between product capabilities and buyer outcomes. A product profile says what it does. A solution profile says what you can achieve with it.

## Messaging Blocks

### Overview

[Instructions:
Establish the problem this solution addresses and the primary outcome it delivers. Problem Space should name the specific operational or business challenge — not a market category, but a real pain that a defined persona owns. Primary Outcome should be a measurable result, not an aspiration.]

[Tips:
- Problem Space should map directly to a challenge in at least one persona profile
- Primary Outcome should be something a customer could verify after deployment — "reduced mean time to remediation from 45 days to under 72 hours" not "improved security posture"]

[Format:
- **Scope:** [organizational | department | team | cross-functional | individual]
- **Theme:** [efficiency | savings | growth | compliance | replacement | consolidation]
- **Problem Space:** [the specific operational or business challenge this addresses]
- **Primary Outcome:** [the measurable result customers achieve]]

### Use Case

[Instructions:
Describe the scenario from the buyer's perspective — where they are today, what's broken or insufficient, and what changes when this solution is in place. This is the narrative version of the Overview — it puts the reader in the situation.]

[Tips:
- Before State should feel recognizable to the target persona — they should see themselves in it
- The Problem should connect to urgency — why can't this wait?
- After State should be concrete, not aspirational — what's actually different day-to-day]

[Format:
**Before:** [the current state — what the team is doing today and why it's insufficient]

**Problem:** [what's breaking, what's at risk, or what's being lost]

**After:** [what changes when this solution is deployed — the new reality]]

### Unique Approach

[Instructions:
How this solution works and what makes it different from alternative approaches to the same problem. This should connect to specific product capabilities and trace back to differentiators in position.md.]

[Tips:
- Lead with the approach, not the product — "we solve this by [method]" not "our product has [feature]"
- If this solution spans multiple products, explain how they compose — the integration is often the differentiator
- Connect to at least one key differentiator from pitch.md]

[Format:
1-2 paragraphs describing the approach, followed by:
- **Key differentiator:** [which pitch.md differentiator this solution makes tangible]
- **Enabling capabilities:** [specific capabilities from product profiles that power this approach]]

### Components

[Instructions:
The products and specific capabilities that comprise this solution. This makes the connection between the outcome (what the buyer gets) and the portfolio (what delivers it) explicit.]

[Format:
| Product | File | Role in Solution | Key Capabilities Used |
|---------|------|------------------|-----------------------|
|         |      |                  |                       |]

[Tips:
- Role in Solution describes what each product contributes — "primary detection engine," "integration layer," "reporting and visualization"
- For single-product solutions, this table may have one row — that's fine, it still makes the mapping explicit]

### Value Delivered

[Instructions:
The measurable outcomes this solution delivers — both quantitative (metrics, percentages, time savings) and qualitative (operational improvements, team experience, risk reduction). These should be grounded in real customer results where possible.]

[Tips:
- Quantitative measures should include ranges or benchmarks when available — "40-60% reduction in MTTR" is more credible than "significant improvement"
- Qualitative measures should describe observable changes — "security team shifts from reactive triage to proactive risk management" not "improved security"
- If proof is thin, state expected outcomes honestly and note that they're projected, not proven]

[Format:
**Quantitative**
- [metric] — [expected range or benchmark]

**Qualitative**
- [observable outcome]]

### Value Messages

[Instructions:
3-5 specific messages conveying why this solution is different and what unique value it delivers. These should be usable in content as-is — grounded in the use case, approach, and value delivered above.]

[Tips:
- Each message should trace to a specific element above — a use case scenario, a unique approach, or a value metric
- Messages should work at the solution altitude — outcomes and approach, not product features
- At least one message should address why existing approaches to this problem fail (connects to Use Case "Problem")]

[Format:
Numbered list of 3-5 messages]

### Personas Aligned

[Instructions:
Which personas this solution is most relevant to and their role in adopting or benefiting from it.]

[Format:
| Persona | File | Role in This Solution |
|---------|------|-----------------------|
|         |      |                       |]

[Tips:
- Role in This Solution should describe how this persona interacts with the solution — "primary user and daily operator," "executive sponsor who approves budget," "technical evaluator who validates architecture"]

### Proof Points

[Instructions:
Customer examples, deployment metrics, or testimonials that demonstrate this solution delivering the outcomes claimed above. Reference quotes where they're solution-specific.]

[Tips:
- Proof should directly support the Value Delivered metrics — if you claim 40-60% MTTR reduction, the proof should show a customer who achieved it
- For new solutions with thin proof, note what exists and what's in development
- Distinguish between named proof (customer willing to be referenced) and anonymous proof (metrics without attribution)]

[Format:
Bulleted list — each proof point with source or attribution]
